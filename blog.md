# Fraud Hunter: Training LLMs to Investigate Government Fraud

Most RLVR environments train agents on puzzles — Sudoku, Wordle, math competitions. The agent reads a prompt, emits an answer, gets a binary score. Clean, fast, easy to iterate on. But real-world reasoning doesn't look like that. It looks like an analyst staring at six database tables, three PDF scans, and a shell company registry, trying to decide whether a dead patient's Medicare claim is fraud or a paperwork error.

**Fraud Hunter** is our attempt to build that analyst — an OpenEnv-compliant RL environment that trains LLM agents to do the kind of multi-step, multi-modal investigative reasoning that FBI, DOJ, and OIG fraud examiners do every day.

---

## The Problem

Investigating False Claims Act violations is genuinely hard. A fraud analyst must:

1. Read a whistleblower dossier and form a hypothesis.
2. Navigate structured CMS Medicare claims databases with millions of rows.
3. Cross-reference corporate registry filings to trace shell company ownership chains.
4. Pull scanned CMS-1500 claim forms, OCR them, and reconcile field-by-field against the database records.
5. Build a formal proof chain: extracted entity → confirmed shell link → documented contradiction → case submission.

The failure modes are also real. Hallucinate a provider NPI and you've just blown your case. Query the same table twice without learning anything and you've wasted your step budget. Submit without a proof chain and the grader dismisses you.

Training an LLM to reliably handle all of this requires an environment that matches that complexity exactly.

## Environment Design

### Synthetic Cases, Real Structure

Each episode is backed by a per-case SQLite database generated by `case_compiler.py`. The schema mirrors CMS SynPUF:

- **Medicare core**: `beneficiary_summary`, `carrier_claims`, `inpatient_claims`, `outpatient_claims`, `prescription_drug_events`
- **Corporate overlay**: `corporate_registry`, `government_contracts`, `general_ledger`, `referral_payments`
- **PPP fraud signals**: `loan_applications`, `payroll_records`, `foreign_affiliations`
- **Multi-modal evidence**: `evidence_documents` (scanned CMS-1500 forms), plus `intercepted_comms/` email threads

The data isn't random. Dollar amounts follow Benford's Law. Provider NPIs pass the Luhn check. PDF scans have injected OCR noise (`0 → O`, `1 → l`) to force the agent to clean the data rather than trust it blindly. Shell company chains are planted at specified depths per tier.

The ground truth — real entities, real shell links, real billing contradictions — lives in a `ground_truth` table the agent can never directly query. The grader reads it; the agent has to earn the same conclusions through investigation.

### 12 Fraud Typologies

Cases are generated across 12 typologies, with per-typology reward multipliers scaled to their real-world complexity:

| Typology | Multiplier |
|---|---|
| Dead patient claim | 1.0× |
| Duplicate billing | 1.0× |
| Upcoding | 1.2× |
| Unbundling | 1.3× |
| Phantom beneficiary | 1.4× |
| Cost/pricing fraud | 1.4× |
| Off-label marketing | 1.5× |
| Product substitution | 1.5× |
| Anti-Kickback Statute (AKS) violation | 1.6× |
| PPP fraud | 1.8× |
| Foreign affiliation concealment | 2.0× |

### 5-Tier Adaptive Curriculum (RLVE)

The difficulty manager tracks a 20-episode rolling reward average per agent session and escalates tiers automatically:

- **Tier 1** — single typology, 2 entities, no shell company
- **Tier 2** — dual typology, 4 entities, 1-layer shell
- **Tier 3** — AKS + 2-layer shell, 6 entities, red herrings planted
- **Tier 4** — PPP + healthcare crossover, 3-layer shell, 8 entities, 3 typologies
- **Tier 5** — multi-sector, 4-layer shell, 10 entities, 5 typologies, OCR + intercepted comms required

The agent is never allowed to plateau. If its rolling average crosses the tier threshold, the next `reset()` draws from a harder pool.

## The Grader: RLVR Without an LLM Judge

The 7-layer grader is fully deterministic. No sampling, no GPT-4 scoring, no human labels. The database is the source of truth.

### Layer Order Matters

```
Layer 1  Schema gate      Invalid JSON → -10, episode ends immediately
Layer 2  CoT presence     Missing <think> block → -2 (soft, non-terminal)
Layer 3  CoT grounding    Real entity names in reasoning → +1
Layer 4  Length schedule  Verbose early steps penalized at -0.005/token (phases out at step 20)
Layer 5  Duplicate query  Same query issued twice → -5
Layer 6  NPI validation   Exact 10-digit Luhn match → +25 / mismatch → -20
Layer 7  Typology matrix  Correct contradiction × typology multiplier
```

The fatal layer is Layer 1. A single malformed JSON action ends the episode immediately with a -10 penalty. This forces agents to learn strict output formatting before they can even attempt to learn investigation strategy — a natural format-first curriculum.

### Reward Structure

```
Case won (full proof chain)        +1000 × 1.5 proof-chain multiplier
Case partial (≥50% coverage)         +250
Contradiction confirmed              +100 × typology multiplier
PDF-backed contradiction             additional × 1.5 document multiplier
Shell link confirmed                  +50
Entity extracted                      +10
CodeAct SQL result                     +5
Per step (time pressure)             -0.1
```

A perfect Tier 1 run (full chain, minimal steps) lands around **+1,480**. A hallucination-heavy submission with no proof chain lands near **−20**.

## Action Schema

Every action must include a `<think>...</think>` block. Skipping it costs −2 and trains away sloppy reasoning from the start.

The agent has 10 action kinds:

| Action | Purpose |
|---|---|
| `sql_query` | Raw `SELECT` on the case database |
| `code_act` | Sandboxed Python with `conn`, `pd`, filesystem helpers |
| `query_corporate` | Entity lookup: name, tax ID, NPI, parent chain |
| `query_medicare` | Beneficiary or claim record lookup |
| `extract_entity` | Formally flag a fraudulent entity (NPI validated) |
| `link_shell` | Assert UBO ownership between two entities |
| `claim_contradiction` | File a billing anomaly with evidence references |
| `ocr_document` | Extract text from a scanned CMS-1500 PDF |
| `compare_doc_vs_claim` | Reconcile OCR fields against the database claim record |
| `submit_case` | Terminal action — seek conviction |

The `code_act` sandbox runs in a worker thread with a 5-second timeout and a blocklist that strips `__import__`, `exec`, `eval`, `os`, `sys`, and all dunder attribute access via AST inspection. Output is capped at 4,096 characters.

## Training: GRPO + DAPO + Zero KL

We trained a Llama-3.2-3B model with a LoRA adapter (rank 16) using `trl.GRPOTrainer` with three specific modifications from the hackathon RL rubric:

**DAPO loss** normalizes by active tokens rather than sequence length. Standard GRPO penalizes short completions even when they're correct — DAPO removes that length bias from the policy gradient.

**Clip-higher** uses an asymmetric PPO clip: ε=0.2 lower, ε_high=0.28 upper. The model is more willing to increase probability on high-reward tokens than to decrease it on low-reward ones, which encourages exploration without destabilizing the policy.

**Zero KL penalty** (β=0) removes the anchor to the base model entirely. This is safe here because the format gate + RLVR grader are the reward signal — reward hacking is structurally prevented by the environment, not by KL regularization.

The reward function replays each completion against a fresh environment instance, accumulates step rewards, and returns a single scalar per completion. Training ran on a single T4 GPU for one epoch across 1,025 cases (205 per tier).

## Real-Time Online RL at Serve Time

Beyond offline training, the deployed server includes a lightweight **contextual bandit** (`OnlineRLPolicy`) that runs at inference time — no GPU required.

The bandit selects the *strategy* for each step — call the LLM, run a SQL schema probe, OCR a document, or list intercepted emails — based on a 10-dimensional feature vector derived from the current observation: step progress, budget remaining, difficulty tier, user intent signals.

After each episode, the client sends the final reward back via `POST /fraud_hunter/online_rl/update`. The bandit applies a one-step policy gradient update instantly:

```
advantage = reward − baseline
w[arm] += lr × advantage × (1[chosen] − prob[arm]) × features
```

This adapts arm selection probabilities to new domains in real time without touching the LLM weights. On a fresh deployment where the LLM hasn't seen certain typologies, the bandit shifts toward exploration arms until the LLM starts winning cases.

## Results

| Policy | Mean Episode Reward | Agentic Recall |
|---|---|---|
| Random baseline | −73.05 | 0.23 |
| Scripted expert (ground-truth-aware) | −5.50 | 0.33 |
| GRPO-trained agent (1 epoch, T4) | *in progress* | — |

The scripted expert's near-zero reward (−5.50) reflects the step-decay cost of 55 steps at −0.1 each — it solved the case but used every step in its budget. The trained agent is expected to submit earlier with a tighter proof chain.

## What This Is, Really

Fraud Hunter is a bet that **verifiable enterprise reasoning** is the next frontier for RLVR. The algebra, coding, and math domains have shown that LLMs can be trained to reason correctly when correct reasoning is checkable. Government fraud investigation is also checkable — the database says whether the entity exists, whether the NPI is valid, whether the claim was filed after the patient's recorded death date.

The environment is the judge. The agent just has to learn to read the evidence.
